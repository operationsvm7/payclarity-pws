## Architecture

**PayClarity PWS** is a commission management SPA deployed on **Vercel**, using TanStack Router (React 19, no SSR) with **Supabase** (PostgreSQL + Auth) as the backend.

> Migrated in June 2026 from Lovable to a standard Vite SPA by removing `@lovable.dev/vite-tanstack-config`, `@cloudflare/vite-plugin`, `@tanstack/react-start`, and `nitro`. Entry point is `index.html` → `src/main.tsx`. `wrangler.jsonc` is kept for reference only — Cloudflare deployment is inactive.

### Routing & Entry Points

- `src/main.tsx` — creates `QueryClient` (stale 5 min, 1 retry) + TanStack Router instance; mounts React at `#root`
- `src/routes/__root.tsx` — root layout; `beforeLoad` enforces auth guard (unauthenticated → `/login`; `status !== 'active'` or `role === null` → sign out); wraps outlet in `QueryClientProvider` + `AuthProvider`
- `src/routes/index.tsx` — renders `<CommissionTool />`
- `routeTree.gen.ts` — auto-generated by TanStack Router Vite plugin on every `dev`/`build`; **do not edit manually**
- `src/router.tsx` — exports `getRouter(queryClient)` factory; SSR-era remnant, not called by `main.tsx`

### Authentication

- `src/lib/auth-context.tsx` — `AuthProvider` / `useAuth()` provides `{ user, profile, loading, signOut, refreshProfile }`
- **Profile fields:** `id`, `email`, `full_name`, `role` (`admin|rep|accountant|null`), `status` (`pending|active|rejected`)
- **Register flow:** email + password + company invite code → `verify_invite_code()` RPC → first user auto-becomes admin; all others start `pending`
- **Admin approval:** Settings → User Management tab (`UserManagementPanel.tsx`) — admin assigns role + sets `status=active`; pending users cannot enter the app
- **Role sync:** `CommissionTool.tsx` reads `useAuth().profile.role` and syncs it into the Zustand store via `useEffect`

Run `supabase/migrations/001_auth_setup.sql` in the Supabase SQL Editor to create `profiles` + `company_config` tables with RLS, triggers, and the `verify_invite_code()` RPC.

### State Management

Global app state lives in the Zustand store (`src/lib/commission-store.ts`, ~1345 lines), **persisted to localStorage**. On first launch with an empty store, demo data is auto-seeded via `loadDemoData()`. The user's `role` comes from Supabase and is synced into Zustand; all other data (agents, invoices, etc.) lives in localStorage only.

Key state categories:
- `agents: Agent[]` — reps with `sponsorId` (downline hierarchy), commission level, wallet
- `invoices: Invoice[]` — sales amount, approval %, charges/credits, CCPF, splits, PDF history
- `financeCompanies: FinanceCompany[]` — dealer/admin fees used in invoice calc
- `personalTiers: PersonalTier[]` / `overrides: OverrideLevel[]` — volume-tiered commission plan
- `splitTemplates` / `splitRules` — multi-participant commission distribution
- `payments`, `disputes`, `adjustments` — payout lifecycle
- `role: Role` — mirrors `profile.role`; controls tab visibility via `AdminGate`

### Business Logic (`src/lib/`)

| File | Key Exports |
|---|---|
| `commission-calc.ts` | `calcInvoice()`, `calcPayouts()`, `rateForVolume()`, `fmtMoney()`, `AgentPayout`, `InvoiceCalc` |
| `ledger.ts` | `buildWallet()`, `buildAllWallets()`, `AgentWallet`, `LedgerEntry`, `explainInvoice()` |
| `generate-invoices.ts` | `buildSaleInvoicePDF()`, `buildAgentCommissionPDF()`, `buildSaleAndDownload()`, `INVOICE_TEMPLATES[]` |
| `templates.ts` | Invoice/email template definitions |
| `i18n.ts` | EN/ES string maps; `language: Lang` in the store toggles the active language |

**Commission math flow:**
1. `calcInvoice(inv, financeCompanies)` → approval amount, charges (dealerFee + CCPF + adminFee), grand total, profit
2. `calcPayouts(agents, invoices, financeCompanies)` → `AgentPayout[]` (personal commission via `rateForVolume()` + downline overrides)
3. `buildWallet(agent, payout, payments, disputes, adjustments)` → `AgentWallet` with full ledger (advances, deductions, tax reserve)

### Core Components

All major UI lives in `src/components/`. Files are large (300–2000 lines) and feature-complete:

| File | Responsibility |
|---|---|
| `CommissionTool.tsx` (~2020 lines) | Top-level shell; `NAV_GROUPS` tab structure; role-based layout; profile menu |
| `ExtraPanels.tsx` (~1419 lines) | Wallet, Simulator, Calendar, Disputes, Templates |
| `SplitsPanel.tsx` (~1102 lines) | Split rules editor (multi-participant commission splits) |
| `CompetitivePanels.tsx` (~670 lines) | Adjustments, CSV Import, Setup Wizard |
| `NewPanels.tsx` (~392 lines) | Dashboard, Reports, Year-End (W-2/1099), Tax Reserve |
| `NotificationsBell.tsx` (~482 lines) | Notification center with audience filtering |
| `InvoiceTimelineDialog.tsx` (~326 lines) | Invoice history with split snapshots |
| `UserManagementPanel.tsx` (~292 lines) | Admin: approve/reject users, assign roles, manage invite code |
| `AdminGate.tsx` | Role-based access control wrapper |
| `ui/` | shadcn/ui primitives (Radix UI + Tailwind); do not modify directly |

`CommissionTool.tsx` is the navigation hub — its `NAV_GROUPS` constant defines all tabs and maps them to panel components.

### Supabase Integration

- `src/integrations/supabase/client.ts` — browser Supabase client (`VITE_SUPABASE_URL` + `VITE_SUPABASE_PUBLISHABLE_KEY`)
- `src/integrations/supabase/types.ts` — TypeScript types from DB schema (`profiles`, `company_config`, RPC functions)
- `client.server.ts`, `auth-middleware.ts`, `auth-attacher.ts` — server-side stubs, unused in SPA mode

Credentials: copy `.env.example` → `.env` and fill in values from the Supabase project dashboard.

### Styling

Tailwind CSS 4 via `@tailwindcss/vite` plugin (no separate `tailwind.config.ts`). Component variants use `class-variance-authority`; classes merged with `cn()` from `src/lib/utils.ts` (`clsx` + `tailwind-merge`). Path alias `@/*` → `src/`. Fonts (Inter, Space Grotesk, JetBrains Mono) loaded via Google Fonts in `index.html`.

### Deployment (Vercel)

`vercel.json` sets build command (`bun run build`), output dir (`dist`), and SPA rewrite (`/* → /index.html`). Add `VITE_SUPABASE_URL` and `VITE_SUPABASE_PUBLISHABLE_KEY` as environment variables in the Vercel project dashboard.
